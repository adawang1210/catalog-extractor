# 磁磚型錄自動擷取 — 實作規劃書 v2.1

> v2 依據第一輪外部技術審查全面修訂;v2.1 整合第二輪審查的五項修補(見 changelog)。**一句話目標**:丟一份 PDF 型錄進系統,輸出裡面每一塊磁磚的「圖片 + 對應規格資訊」清單,且 pipeline 必須是**通用型**(不為任何單一廠商、也不為任何單一「樣本族群」寫死規則)。
>
> **本資料夾(`catalog-extractor/`)即為實作根目錄,後續所有程式碼、普查結果、gold set 皆放這裡。**

## v2.1 相對 v2 的變更（第二輪審查修補）
1. **Stage 3B 跨頁綁定補完整規格**:產品頁的色樣↔色名綁定明定走 VLM(不用「最近圖說」啟發式,避免與 §4.3 矛盾);Stage 1 新增第四類 `product_page`;色名鍵帶系列範圍;純情境照型錄明定「圖片欄位留空」。
2. **fallback 迴路新增文件級健全性檢查**:攔「高信心誤分類」造成的整份靜默漏抓(頁級低信心升級攔不住自信的錯)。
3. **gold set 標註規格補齊**:頁級標籤(M1 驗收依賴)、「一個產品」計數規則、dev/held-out 切分。
4. **無代碼型錄路徑**:普查加測占比;無代碼時替代身分鍵 `(系列, 色名, 尺寸)`;schema PK 改 `(doc_id, code)` 複合鍵。
5. **§8 綁定 spike 新增第四方案「多圖 crop gallery」**(可能最強);**Stage 2B 改以 `get_drawings()` 向量解析為主**、raster CV 為輔。
6. Stage 4 掃描檔路徑改模糊匹配(編輯距離),避免 OCR 噪音灌爆 needs_review。
7. entity resolution 明定範圍:僅同 vendor/同文件內合併。

## v2 相對 v1 的主要變更（changelog）
1. **「附錄」正名為「規格頁（spec page）」**,並明寫規格頁**可散佈全書**,不是只在書尾。v1 的「附錄才是真身」敘事是從 4 份樣本(其中只有 2 份真有書尾附錄)過度歸納。
2. **新增 §0「動工前先決事項」**:corpus 普查、綁定機制 spike、gold set、授權/資料模型/色準決策——這些必須先於 M1。
3. **Stage 1 特徵去擬合**:移除「小色樣」「公制尺寸」「義文關鍵字」等寫死假設,全部改 per-document 統計;關鍵字只加分不減分;尺寸/代碼格式支援多制。
4. **跨頁綁定升為主路徑**(不只可選):處理「規格頁只有文字、色樣在別頁」的型錄。
5. **新增無 XObject 的 CV fallback**:抽不到內嵌圖(向量色塊/整頁大背景)時,render 後用傳統 CV 產候選框。
6. **新增 §8 swatch_id↔VLM 綁定機制設計**,並提前到 spike 階段驗證(v1 把它藏在一句話、排在 M3,是最大技術風險)。
7. **新增 §9 N:M 資料模型與正規化規則**(v1 是 1:1 假象)。
8. **新增 §10 評估方法與 gold set**(v1 用建立假設的同 4 份驗收 = 自我印證)。
9. **fallback 改 page-level 信心分數 + 加入 entity resolution**(逐頁 VLM 會產生重複產品)。
10. **Stage 4 驗證改對回「文字層 token 集合」**而非脆弱的表格結構;camelot 降為可選。
11. **技術選型標注授權地雷**(PyMuPDF AGPL → 評估 pypdfium2)、**色彩管理(ICC/CMYK)**、OcrMac 的 macOS 綁定。
12. **成本估算重做**(納入 fallback),並點明真正的成本主體是 needs_review 人工。

## 0. 動工前先決事項（先於 M1）

這四件事必須在寫 pipeline 主體前完成——它們把「假設」變成「證據」,決定後續投資規模。

### 0.1 Corpus 普查腳本（第一優先）
寫一支輕量腳本掃過全部 300~400 份 PDF,對每份輸出:是否有文字層、頁數、每頁 XObject 數與尺寸分布、是否偵測到向量色塊、spec 訊號命中的頁數與位置分布。**目的**:把下列數字實測出來——
- 掃描檔（無文字層）占比 → 決定 OCR 路徑投資。
- 「無 spec 頁 / 產品直接排頁上（41Zero42 式）」占比 → 決定 fallback 與跨頁綁定投資。
- 總目錄型（>100 頁多系列）占比 → 決定成本與流程假設。
- 向量色樣 vs 內嵌點陣色樣占比 → 決定 §6 Stage 2B 的投資。
- **偵測不到重複短代碼的型錄占比(無代碼型錄)** → 決定 §9 替代身分鍵與 Stage 4/entity resolution 的無代碼路徑投資。

**普查結果(2026-07-08,Eonian 2026 corpus 抽樣,13 家各 1 份,census.py → output/census.csv)**:
- 掃描檔 **0/13** → OCR 只做 Stage 0 偵測+標記,不實作 OCR 路徑。
- 無 spec 訊號頁 **0/13** → spec-page-first 成立;fallback 先只做文件級健全性檢查層。
- 總目錄型 **~15%**(FMG 284 頁、Level 103 頁);**spec 頁均值 ≈22 頁/份(非 4~6 頁)**,763 頁中 289 頁命中 → §12 VLM 呼叫量上修 3~4 倍(約 7,000~9,000 次,金額仍低百美元級)。
- 無代碼 **1/13**(Topcer);且 Topcer 向量填色矩形 1,395 個(全樣本最高)→ 疑為向量色樣型錄,Stage 2B 在 M2 必做。
- **新規則:跨「文件」重複的短 token 是產業規格詞彙(R10/R11 防滑、20MM 厚度、EN14411),不是 SKU,代碼候選須排除**——與 logo「跨頁重複」同邏輯的升級,寫入 Stage 1 特徵與 Stage 4 驗證。真 SKU 樣態:Mosa `5102V`、Marazzi `M6RN`、Sodai `C1378`。
- Vogue、kytile 資料夾無 PDF(待向資料方確認)。

### 0.2 綁定機制 spike（第二優先，最高風險）
在寫任何 pipeline 前,單獨驗證 §8 的 swatch_id↔VLM 綁定機制。用**最壞情況**測:Mosa 密集網格頁、Marazzi 36~60px 小色樣列。**四個候選方案(含多圖 crop gallery)都要進 spike 比較。** 若全數失效,整個 Stage 3 要重設計——所以先知道。

### 0.3 建 gold set 與指標（第三優先）
隨機抽 15~25 家廠商,依 §10 的標註規格(含**頁級標籤**與**產品計數規則**)人工標註,並做 **dev / held-out 切分**。**停止用建立假設的那 4 份當驗收標準。**

### 0.4 動工前要拍板的決策
- **授權/部署棧**:PyMuPDF 是 AGPL-3.0,camelot 依賴 Ghostscript(亦 AGPL)。決策表述為權衡:**買 Artifex 授權(省工) vs 遷移 pypdfium2(省錢但非 drop-in,見 §11)**;若為內部工具不對外分發,AGPL 合規路線可能可行——**先問法務再動手**。OcrMac 綁死 macOS,而營運大概率在 Linux server → OCR 路徑要可移植。
- **資料模型**:先定 §9 的 N:M schema(含複合 PK 與無代碼身分鍵),否則 M3 後重構。
- **色準要求**:向下游確認磁磚顏色的色準需求(ICC/CMYK-aware 轉換 vs naive 轉 RGB)。

## 1. 目標與範圍

**輸入**:300~400 家不同廠商、格式各異的磁磚型錄 PDF,每份約 50 頁(但**普查後**可能發現總目錄型 >100 頁,見 §0.1)。

**輸出**:結構化產品目錄,每個磁磚產品一筆:代碼(SKU)、色名/系列、尺寸(可多個)、表面/技術規格、對應磁磚圖片檔。

**硬性要求**:通用型 pipeline——每步行為由當下文件內容推導,不以廠商名字為 key,也不預設任何「樣本族群」的長相(尺寸制、語言、代碼格式)。企業級:隱私、成本可控、可長期營運。

**明確不做**:不為特定廠商客製腳本;不強求從情境照/藝術拼貼頁解出個別產品(刻意排除)。**推論**:若某型錄除情境照外沒有任何乾淨產品圖(spec 頁純文字 + 全書皆氛圍照),則該型錄的產品**圖片欄位留空**,只出規格——不裁情境照湊數。

---

## 2. 現況與限制

- 環境:macOS,**磁碟吃緊**。
- 既有:Docling + RapidOCR(PP-OCRv4/torch);torch 反覆載入是資源/磁碟壓力主因。
- 核心病灶:pipeline 最初為 **41Zero42（6 份）** 打造,過度擬合其排版;對其他廠商誤抓/漏抓/配對錯。
- 料號 regex 已寫死(`\d{7}` 給 41Zero42、`Exxx` 給 Eureka),每進新廠商就破。

---

## 3. 問題定位

屬於 **Document AI / VDU**,橫跨:電腦視覺(找+分類圖片)、文字抽取(原生抽文字層、掃描才 OCR)、空間語意理解(圖文綁定)。兩個核心難題:**A 圖片誤抓/漏抓**、**B 圖文綁定**。

## 4. 關鍵發現與版面型態分類

### 4.1 四份樣本實測（證據力有限，n 小且全為歐系）

| 型錄 | 產線 | 版面型態 | 色樣尺寸 | 文字層 | 規格資料所在 |
|---|---|---|---|---|---|
| Marazzi「Actually」 | TCPDF 自動生成 | 逐頁表格(色樣左、規格右) | 36~60px | 有 | 各表格頁（非書尾） |
| Mosa「Solids」 | InDesign | 色樣網格 + 代碼在下 | **172~718px（大）** | 有 | 網格頁 p20-21 |
| Heritage | InDesign | 情境照為主 + 書尾規格頁 | 小 | 有 | p11-14 |
| AlterEgo | InDesign | 情境+藝術拼貼 + 書尾規格頁 | 小 | 有 | p23-28 |

**重要更正**:只有 Heritage/AlterEgo 真有「書尾附錄」;Marazzi/Mosa 的規格資料散在全書。因此**不要假設 spec 頁在書尾**,只能假設「頁面內容具有 spec 特徵」。色樣尺寸每家差異極大(36px vs 718px)→ **絕不預設大小**。

### 4.2 核心機制（正名後）
**spec 頁(規格頁)才是產品目錄的資料來源;情境/氛圍頁降級為可選用途圖。** 但 spec 頁**可能出現在任何位置**(書尾、各系列章節結尾、逐頁交錯)。Stage 1 做的是「判斷這頁是不是 spec 頁」,不是「找書尾那一塊」。

### 4.3 過度擬合診斷（保留 v1 結論）
41Zero42 把代碼印在每張磚旁的產品頁上,故「最近代碼」對它有效;主流型錄不這樣,真正配對在 spec 頁。舊 pipeline 是對著 41Zero42 打靶。**引申鐵律:任何「距離最近」式的圖文綁定啟發式都禁用——包括 Stage 3B 的產品頁(見 §6)。**

### 4.4 已知會打破假設的版面型態（普查要量占比）
1. **spec 頁純文字、無色樣**:色樣在前面產品頁、旁標色名 → 必須**跨頁綁定**(§6 Stage 3B)。
2. **無 XObject**:向量色塊、整頁單一大背景圖 → 抽不到內嵌圖 → 需 **向量/CV fallback**(§6 Stage 2B)。
3. **大板(slab)型錄**:產品圖滿版紋理 → 「大滿版=情境照」會誤丟 → 需靠「鄰近是否有代碼/尺寸圖說」修正 + **文件級健全性檢查**兜底(§6 Fallback)。
4. **總目錄型(200~400 頁多系列)**:破壞「每份 spec 頁才 4~6 頁」與成本假設。
5. **無代碼型錄**(純行銷型,只有系列+色名+尺寸):Stage 4 驗證、entity key 都不能假設代碼存在(§9)。

## 5. 架構決策與通用性鐵律

- **spec-page-first(可散佈全書)**:偵測 spec 頁 → 從 spec 頁建目錄;情境頁降級。理由:spec 特徵跨廠商通用、色樣乾淨、跳過最混亂的視覺頁、省 VLM 成本。
- **不走純 YOLO 級聯**:此量級成本非瓶頸,不值得跨 300 家標註訓練。
- **通用性鐵律**:
  - 所有門檻/格式 **per-document 統計**(色樣尺寸取眾數、代碼「短且重複」由該份統計、尺寸 token 支援 `60x120`/`600x600mm`/`12"x24"`/`60×60`/`18,2x21`)。
  - 關鍵字(`Fascia`/`colori e formati`/`IMBALLI`…)**只加分、不減分**;缺某語言對應詞的頁不得因此降級。
  - **多訊號投票**,不靠單一判準(任何單一規則都會被某種排版打敗)。
  - **禁「距離最近」式圖文綁定啟發式**;綁定一律 VLM 或結構性證據(同格、同列、同 entity key)。

## 6. Pipeline 詳細設計

### Stage 0 — 分流（原生 vs 掃描）
文字層檢查;有→關 OCR;無→OCR fallback。

### Stage 1 — 頁面分類（spec / product / lifestyle / other）※通用關鍵
per-document 特徵(全部統計而來,不預設大小/語言/格式):
- **同尺寸圖片重複成陣列/叢集**的密度(取該份眾數尺寸,不設「小」)。
- 短且重複的代碼 token 密度(「短」「重複」由該份統計)。
- 尺寸 token 密度(多制 regex:公制/英制/mm/×/逗號小數)。
- 弱加分:spec 關鍵字命中(多語表,只加分)。
- 大滿版圖 = lifestyle 訊號(但見 §4.4.3 大板例外:若滿版圖旁有代碼/尺寸圖說則不判 lifestyle)。

**四分類**:
- `spec_page`:色樣+代碼+尺寸密集,產品目錄主資料來源。
- `product_page`(v2.1 新增):單頁少量乾淨產品圖 + 鄰近有色名/代碼圖說,但缺尺寸/價格密度——是 Stage 3B 跨頁綁定的圖片來源。
- `lifestyle_page`:情境/氛圍,跳過(可選事後反查)。
- `other`:封面/圖例/版權。

輸出頁級信心分數,供 fallback 用。

### Stage 2A — spec/product 頁抽色樣(有 XObject)
PyMuPDF 抽 XObject;**smask 去重、xref 重複去重、per-page 自適應尺寸/長寬比過濾**;記錄位置與 `swatch_id`。

### Stage 2B — 無 XObject 的向量/CV fallback
spec 頁 XObject 候選為零(向量色塊/整頁大背景)時,依序:
1. **主路徑:解析 PDF 繪圖指令**——`get_drawings()`(或 pdfium path 物件)取**填色矩形的精確邊界**當候選色樣框,比像素級 CV 穩一個量級,對純色/網格色樣尤其如此。
2. **輔路徑:render + 傳統 CV**(連通區域/輪廓)——只在繪圖指令也拿不到候選時用(如整頁被壓成單一大背景圖)。
兩者皆**不使用 LLM 座標**,符合鐵律。

### Stage 2C — 抽文字/表格
pdfplumber 抽文字(原生 100% 準);表格抽取(camelot **可選**,不當驗證依據)。

### Stage 3A — 同頁綁定 + 結構化
spec 頁 render 圖 + 文字 + swatch 清單(帶 id)→ 託管 VLM → 每筆產品 JSON。綁定靠 **swatch_id(見 §8 機制)**,不靠座標;代碼由 VLM 讀,不用固定 regex。

### Stage 3B — 跨頁綁定（v2.1 補完整規格）
適用:spec 頁純文字、色樣在 `product_page` 上。流程:
1. **product_page 也送 VLM**(與 3A 同機制),輸出該頁的 `{色名/代碼圖說 ↔ swatch_id}` 配對。**不用「最近圖說」啟發式**——那是 §4.3 判死的邏輯;此成本已計入 §12。
2. **系列範圍判定**:用章節標題/目錄頁/頁首頁尾的系列名,判定每頁屬於哪個系列(色名不是唯一鍵,`White`/`Grey` 跨系列大量重複)。
3. **合併**:entity key = 正規化後的 `(系列, 代碼)`,無代碼時 `(系列, 色名, 尺寸)`(§9),把 spec 頁的規格連回 product_page 的 swatch。
4. **無圖可綁**(全書只有情境照):圖片欄位留空,不裁情境照(§1)。

### Stage 4 — 驗證（防幻覺）
VLM 輸出的代碼/尺寸字串,**必須存在於該頁(或關聯頁)的文字層 token 集合**(不要求對回表格結構,避免 camelot 抽壞誤標)。比對前先過 §9 正規化(空白/大小寫/全半形)。**掃描檔路徑改模糊匹配(編輯距離門檻)**——OCR 噪音下的精確匹配會把正確輸出灌爆 needs_review。對不上 → `needs_review`。

### Stage 5 — 輸出 + entity resolution
寫入 DB / 圖床(以 swatch_id 關聯);**fallback/逐頁模式必做產品去重**(entity key 合併)。**合併範圍限同 vendor/同文件內**——代碼跨廠商不唯一,全庫合併必撞。

### Fallback（page-level + 文件級雙層）
- **頁級**:低信心頁**個別升級**為逐頁 VLM(而非整份切換)。
- **文件級健全性檢查(v2.1 新增)**:頁級升級只攔「猶豫的錯」,攔不住「自信的錯」(如大板頁被高信心誤判 lifestyle → 整份靜默漏抓)。整份跑完後,若**產品數相對於頁數/尺寸 token 命中數低得離譜**(例:50 頁型錄僅抽出 0~2 個產品,但全書尺寸 token 命中數十處),整份升級重跑逐頁 VLM。
- 逐頁模式產生的重複產品用 entity resolution 合併。成本見 §12。

## 7. 圖片分類策略

**v1 只做第一層便宜訊號;CLIP/DINOv2 移 backlog**(spec/product 頁由 VLM 順便分類、lifestyle 頁跳過,故第二層 v1 幾乎用不到;且 36~60px 裁切放大到 224² embedding 品質差)。

**第一層(便宜訊號,多訊號投票)**:尺寸/長寬比、**跨頁重複(xref 或 pHash)→ logo/裝飾**、頁面型態(色樣只在 spec/product 頁找)、近單色/帶 smask/極小 → 濾掉。

**必守的坑**:純像素會把「緊貼地板特寫」與「色樣」搞混 → **視覺 + 尺寸 + 頁位一起判**;logo 用「跨頁重複」判,不用視覺模型硬猜。

**backlog(量大或 fallback 需要時)**:MobileCLIP(Mac 輕量)/DINOv2 embedding + 小分類頭。

## 8. 綁定機制設計

> **§0.2 spike 已完成(2026-07-08,詳見 output/spike/REPORT.md):純幾何「反向 token 指派」在四種代表頁型拿到 50/50(100%),零模型零成本。Stage 3 主力定案為「幾何綁定(spike_geom.py 的三規則)+ Stage 4 token 驗證 + 抽取覆蓋率檢查」;VLM 降級為低信心頁仲裁者(Gemini 免費層即可,無付費 API 需求)。本地小 VLM 暫不做。以下 VLM 綁定四方案保留作仲裁者路徑的參考。**

問題:VLM 怎麼知道哪個裁切是哪個 id?四種候選:

1. **Set-of-Mark（疊編號標記）**:在 render 圖上疊數字標記。痛點:Marazzi 色樣 36~60px,標記比色樣大;Mosa 一頁幾十格,密集標記易對錯號。
2. **文字座標描述**:把每個 swatch 的相對座標以文字給 VLM(「編號 3 位於左上,x∈…、y∈…」),不疊圖。避免遮擋,但 VLM 對「座標↔畫面位置」對應力有限。
3. **分區放大分批送**:把頁面切成幾塊、每塊放大後標記較少的 swatch,分批送。準度高但呼叫數增、需還原。
4. **多圖 crop gallery(v2.1 新增,可能最強)**:把裁好的 swatch 縮圖本身連同編號一起送(`image 1: <crop>, image 2: <crop>, …` + 整頁 render),讓 VLM 做「哪張縮圖對應哪筆規格」的匹配。避開小色樣遮擋問題,也避開文字座標的空間對應弱點。

**spike 驗收**:用 Mosa 密網格 + Marazzi 36px 小色樣列,量四種方案的綁定正確率(需先對這幾頁做少量人工 ground truth)。若都不過關,Stage 3 改採「先 CV 精準裁切 + 純視覺相似度回配」或「純文字層綁定(代碼/色名)」。

## 9. 資料模型與正規化

**N:M schema(非 1:1)**:
- 一張色樣可對應多個尺寸/多筆 SKU(同色多規格共用色樣)。
- 一個系列/色名可對應多筆代碼(不同表面/尺寸各一碼),各可有自己的色樣。
- 建議實體:`Product(doc_id, code) 複合 PK`、`ColorVariant(series+color)`、`Swatch(swatch_id)`、關聯表 `product_swatch(N:M)`、`product_size(1:N)`。
- **代碼跨廠商不唯一**——單欄 code PK 在全庫合併時必撞,一律帶 `doc_id`(或 vendor_id)。
- **無代碼型錄的替代身分鍵**:`(系列, 色名, 尺寸)`;Stage 4 驗證與 entity resolution 改以此鍵運作(占比由 §0.1 普查決定投資)。

**正規化規則(先定,供 Stage 4 比對)**:
- 尺寸:cm/mm/inch 統一;逗號小數 `18,2`→`18.2`;`×`→`x`。
- 代碼:去空白差異(`MC 1G`≡`MC1G`);大小寫;全形/半形。
- 色名:多語別名歸一;**色名鍵一律帶系列範圍**(見 Stage 3B)。

## 10. 評估方法與 gold set

- **gold set**:隨機 15~25 家廠商,人工標註:
  - **頁級標籤**(spec / product / lifestyle / other)——M1 驗收與信心門檻校準依賴它。
  - 每份的產品清單:代碼、尺寸、色名、swatch 配對。
  - **「一個產品」計數規則先寫死**(一個代碼一筆;同色多尺寸依 §9 schema 為一 ColorVariant 多筆 product_size),否則標註者標不一致。
- **dev / held-out 切分**:15~25 家中切出 5~8 家 held-out,**只在里程碑驗收時碰**;每輪調參只用 dev set,否則 gold set 變 dev set、泛化數字虛高。
- **指標**:頁分類 recall/precision、product-level recall / precision、binding accuracy(圖文配對)、欄位級 F1(代碼/尺寸/色名)。
- **流程**:每個里程碑對 gold set 跑,量化「41Zero42 略降、其他拉上來」是否成立;**不再用建立假設的 4 份當驗收**。

## 12. 成本與維運

- **spec 頁主路徑**:300~400 份 × 4~6 頁 ≈ 1,200~2,400 次 VLM 呼叫。
- **product 頁(Stage 3B,v2.1 新增計入)**:僅「spec 頁純文字」的型錄需要;占比待普查,每份估 10~30 頁 product_page,以 20% 型錄計 ≈ 700~2,100 次。
- **fallback(逐頁)**:若無 spec 頁廠商占 ~20%(**需普查證實**),≈ 70 份 × 50 頁 ≈ 3,500 次——**fallback 可能超過主路徑**;總目錄型再加碼;文件級健全性檢查觸發的整份重跑也計入。
- 絕對金額仍低到中百美元級(Batch API 砍半、prompt caching 降 input);**但真正的成本主體是 needs_review 人工**——若 10% 產品進佇列即千筆級人工,要估比例、規劃審核介面。
- 延後優化:表格型 spec 頁文字層已含全資訊,可改純文字 LLM(便宜一個量級);但增加分支,v1 全走 VLM+Batch 較簡單,量大後再做。

## 13. 風險與已知的坑

1. LLM 座標不可靠 → 裁圖靠 XObject/向量解析/CV,不靠 LLM bbox。
2. XObject 抽圖坑:smask/xref 重複/碎片/大背景/向量圖抓不到 → 配去重+尺寸過濾+Stage 2B。
3. 過度擬合(含 v1 自身):禁止「某廠商 / 某樣本族群長相」的寫死(代碼/尺寸格式/語言/大小);**禁「距離最近」綁定啟發式**。
4. 色樣 vs 地板特寫混淆 → 視覺+尺寸+頁位一起判。
5. 掃描檔 → Stage 0 偵測切 OCR;Stage 4 用模糊匹配。
6. spec 頁無色樣 → 跨頁綁定(Stage 3B,product_page 走 VLM)。
7. 無 XObject → Stage 2B(向量解析優先)。
8. 大板/總目錄型 → Stage 1 例外處理 + **文件級健全性檢查**兜底 + 成本重估。
9. fallback 重複產品 → entity resolution(限同 vendor/文件)。
10. 無代碼型錄 → 替代身分鍵(§9),普查量占比。
11. 授權(AGPL)、色彩(ICC/CMYK)、可移植性(OcrMac)。
12. 多語言/特殊字元/加密/損毀/旋轉/JPX-JBIG2 檔 → 邊界處理。

## 14. 實作里程碑

- **M0｜先決事項(§0)**:corpus 普查 + 綁定 spike(四方案)+ gold set(含頁級標籤、dev/held-out 切分)+ 授權/schema/色準拍板。**這是動工第一步。**
- **M1｜偵測層**:Stage 0 + Stage 1(四分類、去擬合特徵)。驗收:對 gold set 頁級標籤的分類 recall/precision。

- **M5｜營運化**:Batch、DB/圖床、needs_review 佇列與介面、成本監控。

## SOP_新增辨識(未來要辨識型錄裡的新東西時,怎麼安全地加)

**第一步判斷:新東西的答案在文字裡還是圖裡?**
- 文字裡(代碼、尺寸、色名、價格帶、包裝規格…)→ 現有幾何+文字層路線能延伸,便宜。
- 圖裡(立體圖、鋪貼情境、表面質感…)→ 需要視覺模型,成本跳一級,先確認真的需要。

**三類需求(由易到難)**:
1. **新欄位、來源頁已在處理**:最簡單,多抽一個 token(例:厚度 CM2、價格帶 A\d+
   ——已在候選流裡,分流保留即可)。
2. **新欄位、來源在目前路由到別處/丟掉的頁**:啟用 Stage 1 路由器的對應分支
   (例:系列敘述在介紹頁=SCH-1)。路由器設計本來就是「不丟頁只分流」,分支是預留的。
3. **全新視覺辨識對象**:最難,才需要 VLM/embedding(例:立體圖辨識成產品展示圖)。
   先問下游要不要,再動。

**加新東西的鐵律(照抄本專案已驗證的方法,一條都別省)**:
1. 先定 schema 再寫程式(SCHEMA.md 先行,欄位、體制、PK 都拍板才動手)。
2. 每欄位強制附溯源 {pdf, page, bbox, method}——出錯時能回頭看是哪頁哪個框來的。
3. 開新版本閘控(v_n → v_n+1),舊路徑驗 bit-identical,新舊可 ablation。
4. 新辨識對象要有**自己的** held-out 和人工 GT;不能拿舊指標背書新對象。
5. 守住「帳面會騙人」:上線前人工抽查真實正確率,總體指標只當輔助。
6. 安全網優先:不確定的丟 needs_review,寧可舉手也不自信輸出錯的。
7. held-out 只能燒一次;建批次時改名重複檔要 md5 比對;任何過濾要與路由器相容
   (分流不丟頁,別做一次性過濾——家具過濾誤殺 Millelegni 的教訓)。
8. **任何「讓某欄變乾淨」的修改,發布前必做全量 diff 自檢,專查兩種方向的意外:
   (a) None → 有值(乾淨化把「誠實的空」變成「自信的錯」);
   (b) 佇列項消失(不確定件以「已解決」身分離隊)。**
   清洗只該產生「junk→null」和「junk→更乾淨的值」;冒出任何 None→有值,都要逐筆
   問「這個值的證人是誰、憑什麼現在才出現」。

**存查|「表面進步暗藏退步」已知兩實例(同型陷阱,鐵律 5 與 8 的由來)**:
- 實例一(M2,v1→v2 綁定):bound-ok 帳面 94.8→93.9「退步」實為度量假象,v1 高分
  裡藏著靜默錯綁——**帳面會騙人,人工抽真實正確率才算數**(鐵律 5)。
- 實例二(2026-07-10,D5 色名標籤 v2「單證人補位」):窄帶清洗把衝突一側的圖說
  變 null 後,殘存的 junk 名(Level EDEP/Marazzi 圖說塊/Topcer 技術頁文字,共 6 筆)
  從「color=None+code_color_conflict 在佇列」變成「自信輸出+離隊」——欄位統計
  全面變好,錯值卻從佇列漏出。靠發布前全量 diff 攔下(鐵律 8)。
  **修法=雙軌守門模式**:衝突/不確定「偵測」用清洗前的 raw 訊號(colorRaw),
  「輸出」才用清洗後的乾淨值——證人不因清洗而消失,守恆(佇列 519)精確不動。
  此模式通用:未來任何欄位清洗(尺寸正規化、surface 詞表、翻譯)都照此拆
  「偵測軌 raw/輸出軌 clean」,不確定判定永遠站在 raw 側。

**要不要付費 API / 自訓模型**:只有第三類需求且量大時才需要重算這筆帳。即使到那天,
順序仍是:先用免費/託管視覺模型 spike,確認真的不夠再談自訓。本專案的條件
(資料可上傳、只避開中國模型、無 GPU、無標註人力)在可預見的未來都指向
**「用託管的,別自己扛」**。

### 為什麼敘述欄位需要文字 LLM key(常見困惑,2026-07-09 固化)

**誤解**:以為外觀描述/功能特性/適用空間/應用場景這些欄位「型錄裡沒有,所以要 AI
生成」。錯——**這些字型錄裡都有**,就是系列介紹頁那幾段文字;抓字用文字層 100%
拿得到,抓字這步不需要任何 key、不花一毛錢。

**key 的真正用途在「抓到之後」**:文字層給的是一整段混在一起的自然語言——一句話裡
可能同時講了外觀(「溫暖的木紋質感」)、空間(「適合浴室與廚房」)、場景(「營造
北歐風住宅」)。而 schema 要的是分好格子的四個欄位。把「一坨連續文字」**讀懂語意、
正確拆進 appearanceDesc / features / applicableSpaces / applicationScenes**,這個
「分欄/歸類」動作才需要文字 LLM(Gemini)。

**一句話比喻:抓字=影印(免費,文字層);分欄=讀懂意思再分類(需 LLM)。**

**為什麼不寫死規則來分**:每家型錄寫法不同——場景有的在標題、有的埋在段中、有的
整段義大利文。寫死「第幾句是場景」「看到 bagno 就是浴室」是對樣本族群過度擬合,
違反本專案通用性鐵律;讓 LLM 讀語意再分類才是通用解。

**為什麼要人本人去開 key、而且現在就開**:申請 API key 只能由本人在
aistudio.google.com 操作(登 Google 帳號、按同意),程式代勞不了。產線做好、要填
這四欄(ROADMAP 的 F)時就會用到——先備著,等待不佔工期、不卡產線。用**免費層即可**:
量小(每系列幾段文字)不花錢,且符合本專案條件(資料可上傳、避開中國模型)。
